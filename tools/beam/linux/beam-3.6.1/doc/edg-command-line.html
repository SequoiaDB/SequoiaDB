Command Line Options of the EDG Parser

The EDG parser supports the following command line options which you might find useful.

Note that EDG specific command line options need to be prefixed with --edg=. For instance: 

--edg=--no_warnings

This is necessary so that there is no conflict between EDG's command line options and the options for your native compiler.

--absolute_file_name_in_diagnostics

(C, C++, Java)

Display absolute file names in diagnostic messages.

--brief_diagnostics

(C, C++, Java)

The diagnostic message is not wrapped when it is too long to fit on a single line. Additionally, the original source line is not displayed.

--diag_suppress

(C, C++, Java)

This option takes a comma-separated list of error numbers: 

--diag_suppress=num,num,...

Suppress diagnostic messages for the specified numbers. Note, that only warnings and informational diagnostics can be suppressed this way. True errors cannot be suppressed. The numbers for each diagnostic message can be displayed using the --display-error-number option (see above).

--display_error_number

(C, C++, Java)

Display the message number in any diagnostic messages that are generated. This option can be used to determine the number to use when suppressing a diagnostic.

--list_macros

(C, C++)

EDG will display a list of all macros that are defined once the source file is parsed. This implies no compilation, and no BEAM analysis - the file is preprocessed only, and only macros are written to the output file (or stdout if no -o is given).

--no_anachronisms

(C++)

Advise the C++ frontend not to accept certain anachronisms that predate the C++ standard and were accepted by C++ compilers at the time. You may wish to experiment with this command line flag if you get unexpected parser messages.

--no_warnings

(C, C++, Java)

Suppress warnings. This is the default. Errors are still issued.

--no_wrap_diagnostics

(C, C++, Java)

The diagnostic message is not wrapped when it is too long to fit on a single line. In other words, the message appears on a single line. This comes in handy if you want to postprocess parser messages.

--preinclude
--preinclude_macros

(C, C++)

These options both take a filename as an argument: 

--preinclude=filename
--preinclude_macros=filename

Include the source code of the indicated file at the beginning of the compilation. This can be used to establish standard macro definitions, etc. The file name is searched for in the directories on the include search list. When the --preinclude_macros variant is used, only the preprocessing directives from the file are evaluated. All of the actual code is discarded. The effect of this option is that any macro definitions from the specified file will be in effect when the primary source file is compiled. All of the macro-only files are processed before any of the normal preincludes. Within each group, the files are processed in the order in which they were specified.

--report_gnu_extensions

(C, C++)

Display information when certain GNU extensions are in use. This is helpful for those who wish to port code from a GNU compiler to a compiler that doesn't support some or all of the GNU extensions. This also enables warnings in general, which are off by default.

More information is here.

--warnings

(C, C++, Java)

Issue warnings and errors. Warnings are disabled by default.

--xref

(C, C++)

This option takes a filename as an argument: 

--xref=filename

Generate cross-reference information in the file filename. For each reference to an identifier in the source program, a line of the form: 

symbol-id#name#ref-code#file-name#line-number#column-number#symbol-kind#symbol-type

is written, where ref-code is D for definition, d for declaration (that is, a declaration that is not a definition), M for modification, A for address taken, U for used, C for changed (but actually meaning "used and modified in a single operation," such as an increment), R for any other kind of reference, or E for an error in which the kind of reference is indeterminate. symbol-id is a unique decimal number for the symbol. The fields of the above line are separated by '#' characters. symbol-kind is self-explanatory. symbol-type indicates the type of the symbol. A '-' sign indicates that no type is available. The following values are possible:

symbol-type Meaning symbol-type Meaning
void a void type func a function type
int an integer type array an array type
fxp a fixed-point type class a class type
float a floating-point type struct a struct type
imag an imaginary type union a union type
complex a complex type ptm a pointer to member
ref a reference type tp a template parameter
ptr a pointer type
