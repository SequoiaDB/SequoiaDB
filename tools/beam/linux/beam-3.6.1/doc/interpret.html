How to Interpret Complaints

Complaints come in two flavors. The simpler ones are of the form 

-- WARNING2     /*not needed*/     >>>WARNING2_foo_5a83512bd36394
"f3.c", line 6:  variable `b' is assigned, but never used

The complaint consists of two lines. The second line provides a location in your source code and a hopefully self explanatory description of the problem. If it is not self explanatory then you can look up the explanation using the complaint name (here WARNING2) in the description of built-in checks. Everything on the first line following WARNING2 can be used for declaring the complaint innocent.

The more complicated flavor of complaints contains a path through the program and has this form: 

-- ERROR2    /*uninitialized*/     >>>5_foo_9269b7a7e84a25
"f3.c", line 12:  uninitialized `c'
ONE POSSIBLE PATH LEADING TO THE ERROR:
 "f3.c", line 6: allocating `c'
 "f3.c", line 9: the if-condition is false
 "f3.c", line 12: getting the value of `c'

The first two lines have the same meaning as before. What follows is an execution path showing the conditions under which the problem can occur. This flavor of complaint is issued only for those problems that occur when a particular execution path is followed, but not otherwise. The cause of the problem could be anywhere along the path:

  • maybe `c' should have been initialized on line 6,
  • maybe the condition on line 9 is wrong,
  • maybe some variable other than `c' should have been used on line 12,
  • maybe something is missing along the path.

It is up to you to figure out what is wrong. BEAM tries to help by listing key decision points along the path. We strongly recommend that you use the emacs interface which makes it much easier to isolate the cause.
